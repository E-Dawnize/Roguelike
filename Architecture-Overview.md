# Unity Roguelike MVVM+DOTS+DI架构总体设计

## 项目概述

这是一个个人学习项目，旨在重构传统Roguelike游戏为现代化架构，结合MVVM（Model-View-ViewModel）、DOTS（Data-Oriented Technology Stack）和依赖注入（DI）三大技术栈。项目目标不仅是实现功能，更重要的是展示**技术深度**、**架构设计能力**和**工程化思维**，适合作为高级工程师的简历展示项目。

## 设计哲学

### 1. 关注点分离原则
- **Presentation Layer**：UI与用户交互，使用MVVM模式
- **Application Layer**：业务逻辑与状态管理，使用Controller-Model模式  
- **Domain Layer**：核心游戏逻辑与高性能计算，使用DOTS ECS
- **Infrastructure Layer**：技术基础设施，使用DI容器管理依赖

### 2. 渐进式架构
- 支持从传统MonoBehaviour逐步迁移到新架构
- 允许混合使用不同技术栈，按需引入
- 保持向后兼容，降低迁移成本

### 3. 技术深度展示
- 体现现代Unity架构的最佳实践
- 展示性能优化、内存管理、可扩展性设计
- 包含完整的设计文档和改进路线图

## 各层职责详解

### 1. Presentation Layer（表示层）
**核心职责**：处理用户界面和用户交互

**组件**：
- **View**：Unity UI组件（UGUI/UI Toolkit），仅负责显示和用户输入收集
- **Binding Manager**：集中管理MVVM绑定，处理View与ViewModel间的数据同步
- **ViewModel**：UI逻辑封装，包含命令和可绑定属性

**技术特点**：
- 使用`INotifyPropertyChanged`实现数据绑定
- 支持OneWay、TwoWay、OneTime多种绑定模式
- 命令绑定支持ICommand和普通方法

### 2. Application Layer（应用层）
**核心职责**：处理业务逻辑和状态管理

**组件**：
- **Controller**：协调Model和View，处理业务逻辑
- **Model**：业务数据模型，包含状态和行为
- **EventCenter**：事件发布-订阅系统，实现组件间解耦通信

**技术特点**：
- 基于事件的松散耦合架构
- 清晰的Controller-Model-Event责任链
- 支持跨场景、跨系统的事件通信

### 3. Domain Layer（领域层）
**核心职责**：处理核心游戏逻辑和高性能计算

**组件**：
- **ECS Components**：纯数据组件，定义实体属性
- **ECS Systems**：Burst编译的系统，处理游戏逻辑
- **Bridge Services**：MVVM与DOTS间的桥梁服务

**技术特点**：
- 使用Unity DOTS实现数据导向设计
- Burst编译优化CPU密集型计算
- Entity Component System实现高性能实体管理

### 4. Infrastructure Layer（基础设施层）
**核心职责**：提供技术基础设施支持

**组件**：
- **DI Container**：依赖注入容器，管理组件生命周期
- **Installer System**：基于ScriptableObject的服务注册配置
- **Configuration**：可配置的启动和运行参数

**技术特点**：
- 支持Singleton、Scoped、Transient生命周期
- 基于资源的可视化服务配置
- 完整的启动流程和场景生命周期管理

## 关键技术集成点

### 1. MVVM与DOTS双向集成
**现状分析**：
- 已有：MVVM→DOTS输入流（通过EcsInputBridge）
- 缺失：DOTS→MVVM数据同步（ECS状态变化通知MVVM）

**设计方案**：
- **正向流（MVVM→DOTS）**：Controller → EventCenter → Bridge → ECS Component
- **反向流（DOTS→MVVM）**：ECS System → Event System → Model Update → UI Binding

### 2. 事件驱动架构
**核心设计**：
- 统一事件中心：支持struct事件类型，避免装箱
- 跨层事件传播：Presentation ↔ Application ↔ Domain
- 异步事件处理：支持主线程与工作线程间的事件传递

### 3. 依赖注入系统
**核心功能**：
- 构造函数注入：支持`[Inject]`注解的构造函数选择
- 字段/属性注入：支持Unity对象的依赖注入
- 生命周期管理：Singleton、Scoped、Transient完整支持
- 作用域隔离：全局作用域与场景作用域分离

## 技术栈选择理由

### 1. MVVM选择理由
- **UI逻辑分离**：清晰分离UI显示和业务逻辑
- **数据绑定**：自动化UI状态同步，减少样板代码
- **可测试性**：ViewModel独立于UI，易于单元测试

### 2. DOTS选择理由
- **性能需求**：Roguelike游戏需要处理大量实体
- **现代架构**：数据导向设计是游戏开发的未来方向
- **技术深度**：展示高性能游戏开发能力

### 3. DI选择理由
- **依赖管理**：清晰管理组件间依赖关系
- **可测试性**：支持依赖替换，便于测试
- **生命周期**：统一管理组件创建和销毁
