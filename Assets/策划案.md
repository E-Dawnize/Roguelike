## 核心执行铁则
本流程严格遵循**「先定底层规范→再搭架构骨架→跑通最小闭环→逐个落地系统→填充内容→优化包装」**的递进逻辑，设置不可跳步的阶段验收红线：**前一阶段100%完成并通过验收，绝对不进入下一阶段**，从根源避免应届生项目高频出现的架构返工、代码混乱、烂尾等问题。

## 阶段0：开工前置准备（0.5-1周，必须100%完成再进入下一阶段）
本阶段核心目标：定死所有不可随意变更的基础配置，避免开发中途换版本、改规范、重构资源，浪费大量开发时间。

### 步骤1：锁死Unity开发环境（1天完成）
1.  **安装固定Unity版本**：必须使用 **Unity 2022.3.40f1 LTS**
    - 选型原因：Unity DOTS/ECS 1.0.x稳定版仅完美适配2022LTS，2023/2024版本存在大量API破坏性变更，极易踩坑，绝对禁止使用非LTS版本。
    - 安装模块：仅勾选「Windows Build Support」，安卓、WebGL、iOS等非目标平台模块全部不安装，减少安装体积与兼容性问题。
2.  **创建项目并锁死渲染管线**
    - 项目模板选择**3D Core（URP）**，禁止使用Built-in渲染管线（DOTS Entities Graphics不兼容）、HDRP管线（增加不必要的复杂度）。
    - URP使用默认配置即可，全程不做自定义修改，不涉及任何图形学相关开发。
3.  **导入必选Package并锁死版本**
    打开Package Manager，仅安装以下固定版本的包，多余包体一律不安装：
    | 包名 | 固定版本 | 核心用途 |
    |------|----------|----------|
    | Entities | 1.0.16 | ECS核心功能包 |
    | Entities Graphics | 1.0.16 | ECS实体URP渲染支持 |
    | Burst | 1.8.15 | DOTS性能编译核心 |
    | Jobs | 0.70.0-preview.24 | 多线程Job系统 |
    | Mathematics | 1.3.1 | DOTS兼容数学库，替代Unity默认Vector系列 |
    | Unity Physics | 1.0.16 | ECS兼容物理系统 |
    | Input System | 1.7.0 | 新输入系统，替代旧Input类 |
    | Addressables | 1.21.21 | 统一资源管理系统 |
4.  **配置项目基础设置**
    - Player Settings中将「Api Compatibility Level」设置为**.NET 6**（适配DOTS开发要求）。
    - 关闭VR/AR支持、移动端相关非必要设置，仅保留Windows平台配置。
    - 配置Unity官方标准`.gitignore`文件，避免Git上传冗余缓存文件。

### 步骤2：搭建版本控制环境（半天完成）
1.  本地创建Git仓库，完成初始化提交，配置好`.gitignore`规则。
2.  创建远程仓库（GitHub/Gitee二选一，优先GitHub，秋招简历可直接放链接），将本地仓库推送至远程。
3.  **定死分支管理规范**：
    - `main`分支：仅存放稳定可运行版本，每个里程碑完成后合并一次。
    - `dev`分支：日常开发分支，每日提交代码，提交信息必须清晰说明修改内容。
    - 禁止直接在main分支开发，禁止提交编译报错的代码。
    - 强制要求：每日下班前必须将代码提交至远程仓库，防止代码丢失、版本无法回滚。

### 步骤3：定死项目文件夹结构与代码规范（1天完成）
提前建好全项目文件夹结构，所有代码、资源必须放入对应目录，禁止随意乱放，降低后期维护成本。

#### （1）Assets根目录固定文件夹结构

#### （2）固定代码开发规范（全程严格遵守）
- 命名规范：类名、方法名、公共字段/属性使用大驼峰命名法；私有字段、局部变量使用小驼峰命名法，私有字段加`_`前缀（例：`_playerModel`）。
- 访问权限规范：所有字段、方法必须显式声明访问修饰符（`public`/`private`/`protected`），禁止使用默认`internal`权限。
- 分层规范：严格遵守MVC与ECS分层规则，下层禁止反向调用上层代码，跨模块通信仅允许通过事件中心实现。
- 文件规范：一个cs文件仅定义一个类，文件名与类名完全一致，禁止一个文件内存放多个类。

### 步骤4：预导入核心美术资源（1-2天完成）
提前完成核心美术资源的筛选、导入、整理，禁止边开发边找资源打断开发节奏。秋招项目美术以「够用、风格统一、免费可商用」为核心原则，无需纠结视觉效果。

#### 必选资源清单（Unity Asset Store/Kenney.nl可直接获取）
1.  角色与怪物资源：选择一套风格统一的低模奇幻角色/怪物资源，必须带Idle/Move/Attack/Die基础动画，提前整理好模型与动画资源。
2.  场景与地牢资源：选择带地板、墙壁、门、障碍物的地牢素材包，提前整理好可直接用于房间模板的素材。
3.  UI素材：选择一套完整的奇幻风格UI素材，包含按钮、面板、血条、图标等基础元素。
4.  音效资源：选择免费商用的游戏音效包，覆盖战斗、UI、环境基础音效。

#### 本步骤验收要求
- 所有资源按固定文件夹结构分类存放，无散乱文件。
- 完成玩家角色预制体制作，Animator控制器基础动画状态搭建完成，可在场景中正常显示。
- 基础场景素材整理完成，可快速拼接测试房间。

### 阶段0 最终验收标准（必须全部满足，方可进入下一阶段）
1.  Unity项目可正常打开，所有Package安装完成，无编译报错。
2.  Git仓库搭建完成，初始提交已推送至远程，分支规范明确。
3.  项目文件夹结构全部创建完成，代码规范文档同步完成。
4.  核心美术资源全部导入、分类整理完成，基础可用素材准备完毕。
5.  项目可正常打包Windows版本，无报错、无异常。

## 阶段1：底层地基搭建（第1-2周，仅搭框架，不写任何业务玩法）
本阶段核心目标：完成全项目依赖的底层工具、基础框架搭建与测试，相当于建好房子的地基与钢筋骨架，后续所有业务开发都基于本阶段的成果。**禁止在本阶段编写任何打怪、技能相关的业务逻辑**。

### 步骤1：开发公共基础层（第1周 前3天）
本层是全项目的底层工具，所有上层模块都将依赖，必须优先开发、全量测试通过。
1.  **全局事件中心（最高优先级，第一个开发）**
    - 实现线程安全的发布-订阅模式，支持任意结构体事件的订阅、取消订阅、广播。
    - 核心要求：支持值类型事件，避免装箱拆箱与GC，所有事件使用结构体定义，不使用委托/UnityEvent。
    - 交付物：`EventCenter.cs`单例类，完成测试用例编写，验证订阅-广播流程正常，无内存泄漏。
2.  **通用单例基类**
    - 实现两套单例基类：`MonoSingleton<T>`（继承MonoBehaviour，用于管理器类）、`Singleton<T>`（纯C#单例，用于纯逻辑类）。
    - 核心要求：懒加载、线程安全、防止重复创建、场景切换不销毁。
3.  **通用工具类**
    - `LogTool.cs`：封装Unity Debug日志，支持日志分级、发布版自动关闭日志。
    - `MathTool.cs`：封装常用数学计算方法，兼容Unity.Mathematics库。
    - `ExtensionMethods.cs`：编写常用扩展方法，包含Transform扩展、字符串扩展、枚举扩展等。
4.  **配置管理框架**
    - 实现ScriptableObject配置基类，定义所有配置的加载、获取统一接口。
    - 提前搭建Excel→ScriptableObject导入框架基础，为后续编辑器工具开发预留接口。
    - 交付物：配置基类、配置管理器，完成配置加载、读取功能测试。

### 步骤2：开发引擎封装层（第1周 后2天）
将Unity引擎核心功能封装为统一管理器，避免业务代码中随处调用Unity底层API，提升代码可维护性与扩展性。
1.  **InputManager输入管理器**
    - 基于Unity新Input System，封装玩家移动、攻击、技能、UI操作等全量输入事件。
    - 核心要求：输入与逻辑完全解耦，输入触发后仅通过事件中心广播事件，业务代码仅需监听事件，无需直接调用Input System API。
    - 交付物：`InputManager.cs`单例类，完成输入捕获、事件广播测试，功能正常。
2.  **AudioManager音效管理器**
    - 封装背景音乐、2D音效、3D音效的播放、暂停、停止、音量控制统一接口。
    - 核心要求：支持AudioSource对象池复用，避免频繁创建销毁带来的性能开销，接口简单易用。
    - 交付物：`AudioManager.cs`单例类，完成音效播放、池化复用测试。
3.  **Addressables资源管理封装**
    - 封装资源异步加载、预加载、释放、引用计数管理统一接口。
    - 核心要求：统一管控全项目资源生命周期，业务代码无需直接调用Addressables API，避免资源泄漏。
    - 交付物：`ResManager.cs`单例类，完成资源加载、释放、引用计数测试。
4.  **存档管理器基础框架**
    - 实现二进制/Json双格式的存档读写接口，支持多存档位、基础数据加密。
    - 核心要求：存档接口与业务数据完全解耦，新增存档数据无需修改管理器核心代码。
    - 交付物：`SaveManager.cs`单例类，完成数据存、读、加密测试。

### 步骤3：搭建MVC基础框架（第2周 前2天）
定死MVC三层的基类与通信规范，后续所有业务模块必须继承对应基类，严格遵守开发规范，从根源避免面条代码。
1.  **Model基类**
    - 定义纯数据基类，仅负责数据存储与数据变更事件广播，不包含任何业务逻辑。
    - 核心要求：数据修改必须通过公开方法执行，禁止外部直接修改字段，数据变更必须广播对应事件。
2.  **View基类**
    - 定义UI/表现层基类，仅负责界面展示、元素刷新、用户操作捕获，不处理任何业务逻辑。
    - 核心要求：用户操作仅广播事件，不直接调用Controller或修改Model，仅监听事件刷新界面。
3.  **Controller基类**
    - 定义逻辑层基类，负责接收事件、处理业务逻辑、更新Model数据，承上启下。
    - 核心要求：仅持有Model引用，不持有View强引用，通过事件中心与View层通信。
4.  **UIManager完善**
    - 基于MVC规范，实现UI层级管理、页面生命周期、弹窗栈、页面加载/显示/隐藏/销毁统一管控。
    - 核心要求：UI页面的创建、销毁、层级完全由UIManager管控，业务代码无需关心UI加载细节。
    - 交付物：MVC三层基类、完善后的UIManager，完成测试UI页面开发，验证MVC通信流程完全正常。

### 步骤4：搭建ECS基础环境（第2周 后3天）
完成ECS基础环境搭建，定好组件、系统开发规范，验证ECS实体创建、系统执行、烘焙流程正常可用。
1.  **定死ECS开发规范**
    - Component：纯数据、无逻辑，必须为struct，实现对应接口（`IComponentData`等）。
    - System：纯逻辑、无状态，按功能分组到对应SystemGroup，定好执行顺序。
    - 提前完成Component文件夹分类，如通用组件、战斗组件、AI组件、生命周期组件等。
2.  **实现ECS基础通用组件**
    - 编写核心通用组件：`PositionComponent`、`VelocityComponent`、`RotationComponent`、`TeamTagComponent`、`HealthComponent`、`LifeTimeComponent`、`DestroyTagComponent`。
    - 核心要求：所有组件仅使用值类型，不使用引用类型，避免GC，适配Burst编译。
3.  **实现ECS基础生命周期System**
    - 编写基础核心System：`LifeTimeSystem`（处理实体生命周期）、`DestroySystem`（统一销毁标记实体）。
    - 验证System执行顺序、Burst编译、Job调度正常。
4.  **ECS最小Demo验证**
    - 编写测试System，创建1000个带Position和Velocity组件的实体，实现并行移动，验证ECS+Burst+Job System正常工作，无报错、性能符合预期。
    - 交付物：ECS基础组件与System，测试Demo正常运行，实体创建、更新、销毁全流程正常。

### 阶段1 最终验收标准（必须全部满足，方可进入下一阶段）
1.  公共基础层所有工具类、事件中心全部测试通过，无Bug、无内存泄漏。
2.  引擎封装层所有管理器全部开发完成，接口功能正常，测试用例全部通过。
3.  MVC基础框架搭建完成，三层通信规范明确，测试用例验证通过。
4.  ECS基础环境搭建完成，核心通用组件与System开发完成，最小测试Demo正常运行。
5.  全量代码无编译报错，可正常打包运行，无异常。

## 阶段2：最小可玩闭环验证（第3-4周，核心是跑通，不是做全）
本阶段是项目生死线，核心目标是用最少的代码，跑通「角色移动→攻击→打怪物→怪物死亡→UI刷新」的最小核心闭环，同时验证MVC与ECS跨层通信的可行性，确保架构设计可落地，而非空中楼阁。

> 核心铁则：仅实现最小闭环相关功能，任何与闭环无关的功能（技能、Buff、遗物、随机地牢等）一概不做。

### 步骤1：玩家模块MVC三层落地（第3周 前2天）
1.  玩家Model层：实现玩家数据模型，定义基础属性（血量、攻击、移速），实现数据修改与事件广播逻辑。
2.  玩家View层：实现玩家角色控制、动画播放，以及UI血条、属性展示功能。
    - 角色移动：监听InputManager的移动事件，控制角色移动与动画切换。
    - 血条UI：监听玩家血量变更事件，实时刷新血条显示。
3.  玩家Controller层：实现玩家核心逻辑，监听输入事件、战斗事件，更新玩家Model数据，广播玩家状态事件。
4.  交付物：玩家可通过WASD正常移动，镜头跟随正常，动画切换流畅，血条UI显示正常。

### 步骤2：ECS核心战斗System落地（第3周 后3天）
仅实现与最小闭环相关的ECS组件与System，多余功能一概不开发。
1.  补充战斗核心组件：`AttackComponent`、`DamageEventComponent`、`BulletComponent`、`MonsterTagComponent`。
2.  按执行顺序实现核心战斗System：
    1.  `MovementSystem`：处理实体移动，根据Velocity更新位置，Burst编译+并行执行。
    2.  `CollisionDetectionSystem`：处理子弹与实体的碰撞检测，输出碰撞事件。
    3.  `DamageCalculateSystem`：处理伤害计算，生成伤害事件。
    4.  `DamageProcessSystem`：执行伤害结算，扣减血量，广播受击、击杀事件。
    5.  `DeathSystem`：处理实体死亡逻辑，广播死亡事件，添加销毁标记。
3.  交付物：所有核心战斗System开发完成，单实体移动、碰撞、伤害、死亡流程测试正常。

### 步骤3：基础怪物与极简AI实现（第4周 前2天）
1.  实现1个最简单的近战怪物ECS实体，完成烘焙脚本，可正常转换为实体。

| 2 | 遗物系统 | 第7周 前3天 | 事件中心、技能/Buff系统、玩家数据模型 | 事件驱动的遗物框架，支持配置化，新增遗物无需修改核心代码，拾取、生效、清空逻辑测试正常 |

    - 优先保证算法生成的地图100%合法、可通行，再优化生成效果。
4.  **行为树AI框架**
    - 先实现核心节点（组合节点、装饰节点、动作节点），再将原有怪物AI重构至框架中，验证框架可用性后，再扩展更多AI行为。

## 阶段5：性能优化与Bug闭环（第13-16周，沉淀秋招面试核心数据）
本阶段是拉开与其他应届生差距的核心环节，秋招面试官最关注性能优化能力，且必须有**可量化的数据支撑**，空讲原理没有说服力。

### 步骤1：全场景对象池系统落地（第13周）
1.  **Mono对象池**：针对UI、特效、音效源等高频创建销毁的Mono对象，实现通用对象池，支持预热、复用、自动回收。
2.  **ECS实体对象池**：针对子弹、怪物、AOE区域等高频创建销毁的ECS实体，实现实体对象池，提前预热实体，避免运行时频繁创建销毁的性能开销。
3.  验收标准：所有高频创建销毁的对象全部实现池化，`Instantiate`/`Destroy`调用次数趋近于0。

### 步骤2：DOTS与CPU性能优化（第14周）
1.  **Burst编译全覆盖**：给所有可添加Burst标签的System、Job添加`[BurstCompile]`标签，验证编译正常无报错。
2.  **并行执行优化**：将所有可并行的逻辑，全部改为`ScheduleParallel()`并行调度，最大化利用多核CPU性能。
3.  **逻辑降频与分帧优化**：
    - AI感知、距离计算、视野检测等非高频逻辑，从每帧执行改为固定间隔执行（如每0.2秒执行一次）。
    - 大量实体的更新做分帧处理，避免单帧CPU峰值过高。
4.  **空间分区优化**：实现四叉树/网格空间分区，仅对玩家视野范围内的实体做AI、碰撞更新，大幅降低大场景多实体的CPU开销。
5.  验收标准：通过Profiler测试，同屏100个怪物+100个子弹场景下，CPU占用较优化前降低50%以上，留存优化前后的对比数据。

### 步骤3：内存与GC优化（第15周 前3天）
1.  **GC全流程清零**：通过Profiler抓取所有GC Alloc点，逐个优化消除：
    - 消除字符串高频拼接、装箱拆箱、堆内存临时对象。
    - 用结构体替代引用类型，用对象缓存替代频繁创建的对象。
    - 核心目标：战斗场景每帧GC Alloc趋近于0，全程无明显GC卡顿。
2.  **资源内存优化**：
    - 完善Addressables资源管理，做好资源分组、预加载、自动卸载，解决资源冗余与内存泄漏问题。
    - 对美术资源做压缩、降分辨率处理，控制内存占用，避免不必要的资源加载。
3.  验收标准：游戏全程无GC卡顿，内存占用稳定无泄漏，留存优化前后的对比数据。

### 步骤4：全流程Bug修复与稳定性打磨（第15周 后2天-第16周）
1.  **全流程测试**：从头到尾完整游玩至少10局，记录所有逻辑Bug、边界问题、崩溃问题，逐个修复闭环。
2.  **边界情况处理**：完善玩家卡地形、技能释放异常、Buff叠加异常、存档异常、地牢生成异常等边界问题的处理。
3.  **稳定性测试**：连续运行游戏30分钟以上，验证无崩溃、无内存持续上涨、帧率稳定60帧以上。
4.  验收标准：游戏无核心逻辑Bug、无崩溃、无严重体验问题，全程稳定60帧运行。

### 阶段5 最终验收标准
1.  所有性能优化方案全部落地，有明确的优化前后对比数据，性能达标。
2.  游戏全程无GC卡顿，CPU与内存占用符合预期，无性能瓶颈。
3.  所有核心Bug全部修复，游戏稳定运行，无崩溃、无逻辑漏洞。
4.  完整的性能优化报告整理完成，所有对比数据留存完毕，可直接用于面试。
